# Plan: krunch — a jury-room deliberation engine for a panel of LLMs
_Locked via grill — by Claude + kyle.dougan · hardened via Codex review (Rounds 1–2)_

## Goal
krunch locks a user-assembled panel of LLMs "in a room" to deliberate on an arbitrary text problem until they reach consensus or deadlock. Agents are provider-agnostic HTTP endpoints; one user-assigned seat acts as a neutral **mediator** (jury foreman) who orchestrates rounds, synthesizes discussion, rules on consensus/deadlock, and surfaces clarifying questions to the user. v1 is a pure text-deliberation engine — problem in → live transcript + synthesized verdict out — with three interaction modes governing when the mediator pauses to ask the user questions. Built as a Tauri desktop app (Rust core, Vue 3 UI).

## Terminology
- **`max_rounds` (deliberation cap):** krunch's in-app hard cap on deliberation rounds. Default **8**, validated `1..=64`, user-adjustable, persisted per session in the audit snapshot. (Distinct from the Act-2 Codex review-round cap in `PLAN-REVIEW-LOG.md`.)
- **Seat:** a configured agent occupying either a Panelist or the single Mediator role.
- **Audit snapshot:** the recorded configuration of a session (§7) — an audit trail, not a guarantee of reproducing remote model behavior.
- **Attempt:** one provider call for one seat in one round. A seat's round output is exactly one *accepted* attempt.

## Approach

1. **Rust core owns all provider I/O.** No API key or outbound request ever touches the webview. The webview invokes Tauri commands (`start_deliberation`, `answer_questions`, `abandon`, `list_sessions`, `open_session`, `export_session`) and receives streamed events. Tauri capabilities are locked to exactly these commands; a strict CSP forbids the webview from making its own network requests. **Every mutating command carries the session id + an `expected_state` token; the core holds a per-session state-transition lock and rejects commands whose expected state is stale** — so concurrent/duplicate `answer`/`abandon` cannot race into duplicate rounds or double spend. **`start_deliberation` (which has no session yet) carries a client `idempotency_key`; the core atomically inserts a `Starting` session row under a UNIQUE(idempotency_key) constraint before launching any provider task — a duplicate start collides and returns the existing session instead of spawning a second run.**

2. **Agent abstraction + capability contract.** Define an `Agent` trait: `async fn stream_completion(&self, req, cancel_token, on_chunk) -> Result<Completion, ProviderError>`. Two v1 implementors: `AnthropicAgent` (native Messages API) and `OpenAiCompatibleAgent` (generic `/chat/completions`; covers OpenAI, Ollama, LM Studio, OpenRouter, vLLM, etc. via configurable `base_url`). Because OpenAI-compatible servers vary in SSE framing, auth, system-message support, and finish reasons, each adapter declares a **normalized capability contract** (`supports_system_role`, `sse_flavor`, `auth_scheme`, `finish_reason_map`); a **compatibility matrix** + adapter contract tests (§11) pin the differences. Trait leaves room for a future `CliAgent`. A configured seat = `{ id (UUID), display_name, provider, base_url, model, system_prompt, temperature, sampling_params, credential_ref (opaque keychain-item id), role: Panelist|Mediator }`.

3. **Deliberation state machine.** States: `Configuring → Starting → Running → AwaitingUser → Running … → Finalizing → Converged | Deadlocked` and terminal failure states `Halted | MediatorError | Interrupted | Abandoned`. (`Starting` is the atomic pre-spawn row from §1's idempotency insert.) Each **round** is atomic and transactional:
   a. **Round-start persisted** (round row + `status=running`) before any call.
   b. Fire all panelist completions **concurrently** (`tokio::JoinSet`), each carrying a **cancellation token** and **per-seat budgets** (§8). Streamed chunks are written as **provisional** rows tagged by `(round, seat, attempt)`.
   c. **Per-attempt isolation:** on a transient failure (strict allowlist: connection reset, 429, 5xx, idle-timeout) after partial output, that attempt's provisional chunks are marked **discarded** and a fresh attempt starts (exponential jittered backoff, one retry). Exactly one successful attempt is later marked **accepted**; discarded attempts never mix into the seat's output. Persistent failure → seat **abstained**.
   d. Parse the **last fenced ```json block** from the accepted attempt → validate against a **versioned stance schema** `{ v, stance, confidence (0..=1), agree_with[] (seat UUIDs), open_questions[] }`. `agree_with[]` is validated against the current round roster (unknown/self/duplicate refs recorded as malformed fields, dropped from the graph). Missing/malformed/invalid = no-stance/abstain, flagged, no retry.
   e. Round proceeds if **≥2 panelists** produced a valid stance; else transition to `Halted` (persisted).
   f. Feed the mediator a **bounded structured ledger** (prior rulings, per-seat latest stance, capped excerpts of this round's raw output — never the full unbounded history) + this round's roster. Raw panelist text is wrapped in delimited **untrusted-data blocks**; the mediator's system prompt instructs it to treat that text strictly as evidence, never as instructions.
   g. Mediator streams its summary, then emits a fenced json block validated against a **versioned ruling schema** `{ v, ruling: CONSENSUS|CONTINUE|DEADLOCK, request_user_input: bool, next_focus, questions_for_user[], assumptions[], summary (capped, the persisted ledger source) }`. The `summary` field is the bounded running synthesis that becomes the next round's ledger (§4); it is length-capped and persisted in the `rulings` row. **Semantic validation:** `request_user_input==true` requires a non-empty, bounded `questions_for_user`; field combinations must be consistent with the active interaction mode. Malformed/missing/invalid ruling → `MediatorError` (recorded, halts cleanly).
   h. **Deterministic consensus guard.** `CONSENSUS` is accepted only if, over the **surviving (non-abstained) panelists as denominator**: (1) build an **undirected agreement graph using only reciprocal `agree_with` edges** (A→B counts only if B→A also holds; one-sided claims are ignored), (2) find the largest connected component of that reciprocal graph, and it must cover **≥ quorum fraction** (default ⌈2/3⌉) of survivors, **and** (3) mean survivor `confidence` **≥ floor** (default 0.6). The guard is computed purely over structured fields — **stance prose is never compared semantically** (non-deterministic). If the mediator claims `CONSENSUS` but the guard fails, it is downgraded to `CONTINUE` and the discrepancy is logged. Thresholds are validated defaults, exposed in settings.
   i. **Atomic finalize:** in one transaction, mark the accepted attempt per seat, discard the rest, write parsed stances + ruling, flip round `status=finalized`.
   j. Apply interaction mode (§5). On guard-passed `CONSENSUS` → `Finalizing`. On `CONTINUE` at `round == max_rounds` → `Finalizing` (deadlock). On `DEADLOCK` → `Finalizing` (deadlock).

4. **Context management (bounded ledger-as-bus).** Panelists receive: the problem + all resolved user Q&A + the mediator's bounded structured ledger (capped running synthesis) + this round's focus. Panelists never receive other panelists' full raw prose across rounds (and can't see this round's peers — the round is parallel). The mediator consumes the ledger + capped excerpts, not unbounded text, so its context stays bounded. Problem text and user Q&A live in delimited data blocks; control instructions live exclusively in system messages.

5. **Interaction modes (differ only in *when the mediator surfaces questions*; the round is always atomic).**
   - **1 — Autonomous:** never pause. For **every** open question it declines to ask, the mediator must emit a concrete entry in `assumptions[]`; these populate the verdict's "Assumptions made" section.
   - **2 — Batched:** the mediator sets `request_user_input: true` at a round boundary when accumulated `questions_for_user` are worth interrupting for (may bundle across rounds) → `AwaitingUser`. The boolean is the machine-readable interrupt signal, validated against a non-empty question list.
   - **3 — Interactive:** pause at **every** round boundary where `questions_for_user` is non-empty.
   In modes 2/3, answers are appended to the shared problem context (delimited data) and the loop resumes at the next round.

6. **Final synthesis & verdict (`Finalizing` state).** Synthesis is modeled as a **synthetic finalization round** (a normal round row flagged `kind=finalization`) so its provider call reuses the exact `attempts → chunks` accept/discard lifecycle (§3b–c, §3i) — one mediator synthesis attempt, under mediator budgets, one retry, exactly one accepted attempt. Success → final solution + "Assumptions made" written, then `Converged`/`Deadlocked`. If synthesis fails after retry → terminal fallback: export the durable transcript with a clear "synthesis failed" note rather than fabricating a verdict. Deadlock is reported honestly (unresolved split named), never faked as consensus.

7. **Persistence (SQLite via `rusqlite`/`sqlx`, app-data dir).** Relational: `sessions → rounds → attempts → chunks` (finalization is a `kind=finalization` round, §6, so synthesis attempts/chunks live here too), plus `stances`, `rulings` (incl. the capped `summary` ledger field, §3g), `user_qa`, `error_records`, and an **audit snapshot**. A **single dedicated writer task** owns the DB connection; streamed tokens are batched into **bounded chunks flushed on a short interval** (no per-token write) to avoid lock contention/latency across concurrent seats. **Every chunk/event carries a `(round, attempt) generation fence`; the writer only persists — and the UI only displays — chunks whose fence matches the live generation, so queued pre-cancel/pre-retry chunks are dropped and can never land after `Abandoned` or after a discarded attempt.** Chunks are provisional→accepted/discarded (§3). **Diagnostic `error_records`** persist redacted error class, adapter, HTTP status, retry count, deadline-hit flag, and provider request id where available. The **audit snapshot** stores per seat: provider, `base_url`, model, model revision where the API exposes it, full sampling params, `credential_ref` non-secret identity (never the key), mediator assignment, `max_rounds`, and guard thresholds. **Startup recovery:** any session left `Running`/`AwaitingUser`/`Finalizing` from a crash is marked `Interrupted`; any round left provisional is rolled back to its last finalized round. All terminal states (incl. `Halted`/`MediatorError`/`Interrupted`/`Abandoned`) are re-openable read-only. **At rest:** plaintext SQLite by default (documented to the user), DB file created owner-only; encrypted storage / retention controls are a v2 opt-in.

8. **Budgets, backpressure & cancellation.**
   - **Input budgets:** the problem statement and each user Q&A answer have enforced **input-size limits** at capture; every provider request is assembled under an **explicit total context-token budget** (system + delimited data + ledger + focus). If the assembled request would exceed a seat's model budget, the oldest capped ledger excerpts / Q&A are elided deterministically (recorded), so unbounded problem/Q&A text can't overflow the context window even though the ledger itself is bounded.
   - **Per-seat output budgets:** connect / idle / total deadlines, max-response-bytes ceiling, output-token cap. Exceeding a cap → **deterministic truncation recorded in the transcript**.
   - **Backpressure (lossless):** UI token delivery uses bounded per-session channels with timed coalescing. On overflow, tokens are **never silently dropped**: the store either receives coalesced resync chunks or the UI marks the gap and reloads the **authoritative persisted text** from the DB — the displayed transcript can never diverge from what's persisted.
   - **Cancellation:** `abandon` propagates a cancellation token to every task and in-flight HTTP stream; post-cancel chunks/events are rejected; the in-flight round is marked `abandoned` and the session transitions to the terminal `Abandoned` state, persisted at the cancellation point.

9. **Secrets & credential routing.** Keys in the **OS keychain** via `keyring`, referenced from seat config by an **opaque `credential_ref` (keychain-item id)** — so multiple accounts for one provider/host are selectable and the config/snapshot hold only non-secret identity. Credential routing is **canonicalized key-to-origin binding**: a key is attached only when the request's canonical origin matches the binding; **HTTPS is required except an explicit loopback opt-in** (for local Ollama/LM Studio); **cross-origin redirects are disabled before credentials are attached**; changing a seat's `base_url` origin requires **explicit user confirmation** before any stored key is routed there. Non-secret config (roster, models, temps, prompts, `max_rounds`, thresholds) in a TOML/JSON file under the app-config dir. Keys never in the config file or repo.

10. **Session-creation validation (before any spend).** Reject invalid sessions up front: `max_rounds` within `1..=64`; **exactly one mediator**; **2–6 distinct panelists** (unique seat ids); every seat has a resolvable `credential_ref` + reachable origin (lightweight preflight); guard thresholds in range. Duplicate ids, missing mediator, unsupported roster → validation error, no session row created.

11. **UI — three states of one window** (Vue 3 + Pinia + Tailwind + shadcn-vue; **Pretext** for text measurement to keep multi-card live streaming + virtual-scrolled transcript reflow-free). Every streamed event carries `{ session_uuid, round, seat_id, attempt, seq (monotonic) }`; the store ignores events whose session/round/attempt is stale or whose `seq` is out of order. All rendered Markdown/HTML is **sanitized**.
   - **Setup:** build roster (seats: provider/model/prompt/temp/sampling/credential), assign the **Mediator** slot, pick interaction mode, set `max_rounds`, type problem, Start (runs §10 validation).
   - **Room:** grid of panelist cards streaming live tokens; distinct mediator panel (summary + ruling); round counter / `max_rounds` progress bar; **convergence strip** (per-round stance + confidence + agreement graph); failed/abstained/truncated/retried cards show explicit badges.
   - **AwaitingUser:** modal/panel collects answers; resume.
   - **Verdict / terminal:** for `Converged`/`Deadlocked`, final solution + assumptions + outcome. For `Halted`/`MediatorError`/`Interrupted`/`Abandoned`, a **terminal failure view** stating the reason + last durable round. Both paths offer copy/save of the **single self-contained Markdown export** (header w/ problem, mode, outcome/reason, round count, audit snapshot; verdict-or-reason; full round-by-round transcript with user Q&A inline). Exported fields are **escaped by default**.

12. **Testing (before UI implementation).** Adapter **contract tests** against the capability matrix (malformed/partial SSE, auth variants, missing system-role, odd finish reasons, redirects). **State-machine + persistence tests** (transactional finalize, per-attempt accept/discard, single-writer chunk flush, crash-recovery → `Interrupted`/rollback, cancellation → `Abandoned`, concurrent-command lock, retry classification, schema-version migration). **Consensus-guard tests** (quorum/confidence math over the agreement graph). **Adversarial fixtures** (prompt-injection in panelist output, forged rulings, `request_user_input` w/ empty questions, mode-inconsistent rulings, malformed/oversized JSON, `agree_with` referencing bogus seats).

## Key decisions & tradeoffs
- **General-purpose text engine, not repo-specific (Q1=A).** Sandboxing/file access deferred to v2.
- **Parallel rounds (Q2=C).** Round latency = slowest seat; discrete consensus events; loses emergent turn-taking (v2).
- **Hybrid consensus + hard `max_rounds` cap + deterministic guard over the agreement graph (Q3=C + R1/R2).** Mediator is final arbiter but cannot declare `CONSENSUS` against a failing quorum/confidence check; stance prose is never semantically compared. Default `max_rounds=8`.
- **Neutral mediator, never a participant in v1 (Q4=A).** Participant-toggle is v2.
- **HTTP endpoints, Anthropic + generic OpenAI-compatible, behind a trait + capability contract (Q5=A + R1).** CLI agents slot in later.
- **Fenced-JSON parsing, versioned + semantically-validated schemas (Q7 + R1/R2).** Malformed = abstain (panelist) / `MediatorError` (mediator); no retry storm.
- **Bounded ledger-as-bus context (Q11 + R1).** Fixes unbounded-growth overflow; trades cross-panelist raw recall for bounded cost.
- **Per-attempt isolation + single-writer chunked persistence (R2).** Retries never mix partial output; per-token DB writes avoided; UI overflow is lossless against the persisted source of truth.
- **Transactional rounds, `Finalizing` state, terminal failure states + recovery (Q9 + R1/R2).** Crash/cancel/synthesis-failure all resolve to a defined, exportable terminal state.
- **All I/O in Rust; keychain-item refs, canonical key-to-origin binding, HTTPS+loopback-only, no cross-origin redirects, origin-change confirmation (Q8 + R1/R2).** SSRF/key-exfil blocked; multi-account supported.
- **Concurrency lock + expected-state tokens on mutating commands (R2).** No duplicate rounds/spend from racing commands.
- **Up-front session validation (R2).** No zero caps, missing mediator, duplicate/oversized rosters, or unusable credentials reach spend.
- **Live streaming with bounded coalesced channels (Q8 + R1).** Room feels alive without flooding the webview.
- **SQLite, read-only past sessions, no mid-run resume (Q9).** Resume is v2.
- **Seat count 2–6 panelists + 1 mediator (Q14).**
- **Prompt-injection isolation + sanitized rendering + escaped export + Tauri capability/CSP lockdown + diagnostic error records (R1/R2).**

## Risks / open questions
- **JSON-block compliance on weak local models.** Small models may routinely malform the block → permanent abstain. Mitigation: strict prompt + lenient-but-validating parser; monitor abstain rate; a "reformat your JSON" nudge is a v2 candidate.
- **Consensus-guard tuning.** Quorum fraction + confidence floor defaults need empirical tuning; wrong values → false deadlock/consensus. Ship conservative defaults, expose in settings.
- **Agreement-graph semantics.** Relying on self-reported `agree_with[]` means a model can misreport agreement; the guard is only as honest as the stances. Convergence strip surfaces this to the user; NL cross-checking of agreement is a v2 consideration.
- **Cost/latency of N concurrent seats.** Bounded by budgets + `max_rounds` + seat cap; a live token/cost meter is a strong early add, not yet specified.
- **Keychain UX on first run** (macOS prompts, dev/headless). Needs a clear key-entry settings flow.

## Out of scope (v1 → v2)
- Repo/file access & agentic tools; sandboxing
- CLI agents (trait + capability contract are ready)
- Mediator-as-participant toggle
- Dynamic / round-robin turn structures
- Mid-run resume & adding rounds to a finished session
- Panelists seeing each other's full raw cross-round prose
- NL cross-checking of self-reported agreement
- JSON (non-Markdown) export
- Encrypted-at-rest storage & retention controls (plaintext + owner-only perms in v1)
- Cost/budget meter beyond per-seat budgets + round cap + seat count
- "Reformat your JSON" recovery nudge for non-compliant models
